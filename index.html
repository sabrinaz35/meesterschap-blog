Killian Valkhof

The rule of least power

4 februari 2026

Killian heeft al veel bereikt in zijn leven en heeft een browser gemaakt gericht voor developers genaamd polypane, waar je verschillende frames in een keer kan zien om het jezelf gemakkelijk te maken.

Tijdens de lecture sprak hij over the rule of least power, dit houdt eienlijk in dat je een keuze maakt in een taal die het meest makkelijk is en het meest geschikt is om te gebruiken, dus als css het kan oplossen dan is css het meest geschikt.

Ook had Kilian het over dat er steeds meer ontwikkelingen plaatsvinden en er eigenlijk steeds minder javascript nodig zal zijn. Zo gaf hij als voorbeeld de mogelijkheid om een "switch" bij een input veld en dat sommige browsers er dan ook zo'n toggle button van maken.

Paul Peter Koch

Browser and support detection: Geschiedenis van browsers

19 februari 2026

Peter Paul Koch vertelde veel interessante dingen een ding sprak mij het meest aan. Het klinkt logisch, maar op hedendag denk je er ook niet meer echt over na, hij had het op gegeven moment over verschillende elementen die vroeger gebruikt werden voor het web wat browsers tot op de dag van vandaag nog steeds moeten gebruiken.

Ook sprak hij over diverse browsers vanaf de start van het web tot aan de dag van vandaag, de geschiedni. Wat mij daarvan het meest is bijgebleven is dat een van de eerste web browsers Mosaic voor een radicale innovatie heeft gezorgd en dat is dat zij de eerste waren waarbij je plaatjes kon zien in de browser.

Als laatste sprak hij over veel vaktermen en legde deze ook uit, dit vond ik wel fijn. Want ik dacht een idee te hebben wat de vaktermen betekende, maar ik had het bij het verkeerde eind. Zoals bijvoorbeeld wat een browser daadwerkelijk is. "Een browser is een platform wat met het web connect, hij moet dingen zoals html, css en javascript engine kunnen lezen en kunnen parsen om voor de gebruiker te kunnen tonen."

Wat ik ook wel interessant vond was dat je twee verschillende modellen hebt de DOM (document object model) en de AOM(accesibility object model) en dat het verschil daartussen zit in de toegankelijkheid, de AOM is meer gefocust op de toegankelijkheid. Waae de AOM de aria-labels wel kan lezen als een toevoeging voor de screenreader kan de DOM dat niet.
